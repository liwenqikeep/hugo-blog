## 前言

如果你已经熟悉了 LLM 的基本原理和提示词工程，一定会遇到 LLM 最让人头疼的问题之一：**幻觉**。模型可能言之凿凿地说出错误的事实，因为它只知道训练数据截止日期前的内容。

RAG（Retrieval-Augmented Generation，检索增强生成）就是解决这个问题最主流的方案。它让 LLM 在回答问题时，**先去外部知识库中检索相关信息，再基于检索到的内容生成回答**——通俗地说，就是给 LLM 配了一个搜索引擎，让它能"查资料"再回答。

本文将从原理层面深入 RAG 的核心流程和关键概念，让你全面理解这项技术。

<!--more-->

## 为什么需要 RAG

在聊 RAG 的原理之前，先看一个场景：

```
用户提问："我们公司 2024 年第一季度的营收是多少？"

LLM 直接回答（无 RAG）：
❌ "抱歉，我的知识截止于 2023 年，无法回答这个问题。"
   或者编造一个数字 → 幻觉

LLM+RAG：
✅ 1. 先去公司内部知识库检索"2024 Q1 营收报告"
   2. 找到相关文档片段
   3. 基于检索结果生成回答："根据公司 2024 年 Q1 财报，营收为 12.5 亿元"
```

RAG 的出现解决了 LLM 三个核心痛点：

| 问题 | 传统 LLM | RAG 方案 |
|------|----------|----------|
| **知识截止日期** | 训练数据有时间限制 | 实时检索外部知识 |
| **幻觉** | 可能编造事实 | 基于检索事实生成，大幅降低幻觉 |
| **私有知识** | 无法获取企业内网数据 | 接入企业内部知识库 |

## RAG 核心流程

一个标准的 RAG 系统由三个核心环节组成：

```
                    RAG 完整流程
                          
 ┌─────────────────────────────────────────────────────┐
 │                     用户提问                          │
 └────────────────────┬────────────────────────────────┘
                      │
                      ▼
              ┌───────────────┐
         ┌───→│  检索 (Retrieval)│←──────────────┐
         │    └───────┬───────┘                │
         │            │                        │
         │            ▼                        │
         │    ┌───────────────┐     ┌──────────────────┐
         │    │  增强 (Augment)│     │   文档库/知识库     │
         │    └───────┬───────┘     └──────────────────┘
         │            │                        ▲
         │            ▼                        │
         │    ┌───────────────┐     ┌──────────────────┐
         │    │  生成 (Generation) │     │   向量数据库      │
         │    └───────┬───────┘     └──────────────────┘
         │            │
         │            ▼
         │    ┌───────────────┐
         └────│  用户得到回答   │
              └───────────────┘
```

下面逐一拆解这三个环节。

### 环节一：检索（Retrieval）

检索阶段的目标是从知识库中找到与用户问题最相关的文档片段。这又分为两个子步骤：

**1.1 文档预处理（离线阶段）**

```
原始文档 → 切分（Chunking）→ 向量化（Embedding）→ 存入向量数据库
```

- **切分（Chunking）**：将长文档切成合适大小的片段。切得太碎会丢失上下文，太大会降低检索精度。常见的策略有按段落切分、按固定 token 数切分、或使用语义切分。
- **向量化（Embedding）**：每个文本片段通过 Embedding 模型转换成一个**高维向量**（通常几百到几千维）。这个向量捕获了文本的语义信息——语义相近的文本，其向量在空间中也彼此靠近。
- **存入向量数据库**：将向量和对应的原始文本存入专门的向量数据库（如 Milvus、Pinecone、Qdrant）。

**1.2 在线检索**

```
用户问题 → Embedding 向量化 → 向量数据库相似度搜索 → 返回 Top-K 相关片段
```

当用户提问时：

1. 将用户问题也用同样的 Embedding 模型转为向量
2. 在向量数据库中执行**相似度搜索**（如余弦相似度、欧氏距离）
3. 返回最相关的 K 个文档片段（如 Top-5）

**为什么叫"语义搜索"？**

传统的关键词搜索（如 SQL `LIKE`）只能匹配字面相同的内容。而向量搜索能匹配**语义相近**的内容：

```
用户搜索："怎么提高系统并发能力？"
传统关键词：找不到包含"并发"的文章
语义搜索：能找到"高并发优化"、"性能调优"、"吞吐量提升"等语义相关的文章
```

### 环节二：增强（Augment）

检索到相关文档后，不能直接把一堆原文丢给用户。**增强环节**将它们组装成一个结构化的上下文，与原始问题拼接，形成最终的 prompt。

典型的增强策略：

```
系统：你是一个智能客服助手。请基于以下参考资料回答问题。
如果参考资料不足以回答，请明确说明你不知道。

参考资料：
[1] 公司 2024 年 Q1 财报摘要：营收 12.5 亿元，同比增长 15%...
[2] 2024 年 Q1 产品线增长数据：核心产品线 A 增长 20%...

用户问题：我们公司一季度营收情况如何？
```

增强阶段的设计对回答质量影响很大。常见技术包括：

- **压缩**：对检索到的多个文档去重、排序，只保留最相关部分
- **重排序（Re-ranking）**：用一个更精细的模型对检索结果重新排序
- **窗口扩展**：如果匹配到的片段太短，扩展其上下文窗口

### 环节三：生成（Generation）

最后一步，将增强后的 prompt 喂给 LLM，由它基于检索到的上下文生成回答。

这个过程与普通的 LLM 调用没有本质区别，但因为有检索到的真实信息作为依据，生成的答案**准确性显著提升、幻觉大幅降低**。

## Embedding 模型

Embedding 是整个 RAG 系统的基石。它的作用是将文本转化为语义向量。

```python
# 概念示意（非真实 API）
"今天天气真好" → [0.12, -0.34, 0.56, ..., 0.89]  # 768 维向量
"今天阳光明媚" → [0.14, -0.31, 0.55, ..., 0.87]  # 语义相近 → 向量距离近
"蒙娜丽莎"     → [0.87, 0.23, -0.12, ..., 0.05]  # 语义无关 → 向量距离远
```

衡量向量相似度最常用的指标是**余弦相似度**（Cosine Similarity），取值范围 [-1, 1]：

- 接近 1：方向相同，语义相似
- 接近 0：正交，语义无关
- 接近 -1：方向相反，语义相反

常用的 Embedding 模型：

| 模型 | 维度 | 适用场景 |
|------|------|----------|
| OpenAI text-embedding-3-small | 1536 | 通用英文 |
| OpenAI text-embedding-3-large | 3072 | 高精度需求 |
| BAAI/bge-large-zh-v1.5 | 1024 | 中文场景 |
| 文心一言 Embedding | 384 | 中文场景 |

## 向量数据库

传统数据库不适合做高维向量相似度搜索（"维度灾难"问题），于是出现了专门为向量检索优化的**向量数据库**。

| 特性 | 传统数据库 | 向量数据库 |
|------|-----------|-----------|
| 查询方式 | 精确匹配 / 范围查询 | 相似度搜索 |
| 索引结构 | B+ 树 | IVF、HNSW、Product Quantization |
| 适用场景 | 结构化数据 CRUD | 非结构化数据语义搜索 |
| 代表产品 | MySQL、PostgreSQL | Milvus、Pinecone、Qdrant |

### 主流向量数据库对比

| 产品 | 部署方式 | 特点 |
|------|----------|------|
| **Milvus** | 自部署（K8s/Docker） | 开源自建，功能最全，适合生产 |
| **Pinecone** | 云托管 SaaS | 免运维，上手快，成本较高 |
| **Qdrant** | 自部署 / 云托管 | Rust 实现，性能优异 |
| **Chroma** | 嵌入式 | 轻量级，适合开发和原型 |
| **PGVector** | PostgreSQL 插件 | 无需额外组件，已有 PG 可用 |

## RAG 的进阶策略

基础的 RAG（Naive RAG）——检索 → 增强 → 生成——已经能解决很多问题。但实践中还有更多优化方向：

### Query 重写

用户的问题可能不够清晰，直接检索效果不佳。可以先用 LLM 把问题改写得更适合检索：

```
用户原始问题："那个新功能什么时候上？"
→ 重写为："公司新功能'智能客服系统'的预计上线时间"
```

### HyDE（假设性文档检索）

先用 LLM 根据用户问题**生成一个假设性的理想答案**，再用这个答案去检索。因为假设答案的语义往往比原始问题更接近目标文档。

### 多轮对话中的检索

在连续对话中，需要结合历史对话上下文进行检索，而不是孤立地处理每条消息。

## RAG vs 微调

一个常见的问题是：既然想让 LLM 知道更多知识，为什么不直接微调模型？

| 维度 | RAG | 微调 |
|------|-----|------|
| 知识更新 | 实时更新知识库即可 | 需要重新训练 |
| 幻觉控制 | 基于检索事实，幻觉少 | 仍可能产生幻觉 |
| 成本 | 无 GPU 训练成本 | 需要 GPU 训练 |
| 知识深度 | 依赖检索质量 | 参数化记忆，推理更自然 |
| 适用场景 | 需要实时/私有知识 | 需要改变模型行为/风格 |

**两者并非对立，而是互补**。一个常见的策略是：用微调让模型学会特定领域的对话风格，用 RAG 提供实时准确的知识。

## 总结

RAG 是解决 LLM 知识瓶颈最有效的方案之一。它的核心思想简单而强大：

> **让 LLM 先"查资料"再回答，而不是凭空生成。**

理解 RAG 的三个核心环节——检索、增强、生成——是掌握这项技术的基础。其中 Embedding 模型和向量数据库是基础设施级别的关键组件。

RAG 也并非银弹，检索质量直接决定了回答质量。如何设计合理的切分策略、选择合适的 Embedding 模型和向量数据库、优化检索结果的重排序，都是值得深入的方向。
