## 前言

支付系统的可用性直接关系到业务收入。理想的支付系统应该做到 **7×24 小时不间断服务**——即使在部分机器宕机、甚至整个机房故障的情况下，依然能够对外提供服务。

本文从架构层面讨论支付系统的高可用设计方案，包括冗余部署、流量调度、降级策略和异地多活。

## 三、降级策略

### 3.1 降级分级

```
降级等级矩阵：

┌─────────────────────────────────────────────────────────┐
│  等级     |  动作             |  影响                    │
├─────────────────────────────────────────────────────────┤
│  L0 正常  |  全功能可用         |  无                     │
│  L1 轻微  |  非核心功能降级      |  关闭对账、报表          │
│  L2 中等  |  渠道降级           |  关闭高成本渠道          │
│  L3 严重  |  核心熔断           |  关闭全部支付功能        │
│  L4 灾难  |  全站只读           |  仅可查询历史订单        │
└─────────────────────────────────────────────────────────┘
```

### 3.2 自动降级实现

```java
@Component
@Slf4j
public class DegradationManager {

    private volatile DegradeLevel currentLevel = DegradeLevel.L0;

    public boolean isAvailable(PaymentRequest request) {
        if (currentLevel == DegradeLevel.L3 || currentLevel == DegradeLevel.L4) {
            return false;
        }
        if (currentLevel == DegradeLevel.L2 && isHighCostChannel(request.getChannel())) {
            return false;
        }
        return true;
    }

    public synchronized void upgradeLevel(DegradeLevel level) {
        if (level.ordinal() > currentLevel.ordinal()) {
            log.warn("降级等级提升: {} → {}", currentLevel, level);
            this.currentLevel = level;
        }
    }
}
```

## 五、关键设计要点

| 设计 | 方案 |
|------|------|
| 无状态设计 | 所有服务节点无状态，通过 K8s 弹性伸缩 |
| 数据库高可用 | MySQL MGR + ProxySQL 读写分离 |
| 缓存高可用 | Redis Cluster + 本地缓存兜底 |
| 消息队列高可用 | RocketMQ DLedger 多副本 |
| 配置中心 | Nacos 集群部署 + 本地文件兜底 |
