## 前言

构建分布式系统与构建单体应用的最大区别在于：**你必须假定你所依赖的任何东西都可能失败**。网络会断、机器会宕、磁盘会坏、消息会丢。

本文不讨论某个具体技术，而是总结分布式系统设计的核心思想和通用方法论——这些思想贯穿在所有分布式系统的设计和实现中。

<!--more-->

## 一、分而治之（Divide and Conquer）

分布式系统的核心思想就是分治——**把一个大问题拆分成多个小问题，分别解决，再合并结果**。

```
应用层面的分治：
├── 水平拆分（分片）：按某个维度拆分数据
│     └── 用户表按 user_id % 16 拆到 16 个库
│
├── 垂直拆分（服务化）：按业务领域拆分
│     └── 订单服务、用户服务、支付服务
│
└── 任务拆分（并行计算）
      └── MapReduce：分片处理 → 汇总结果
```

**分治的核心问题：拆分后如何协同？**

| 拆分类型 | 协同方式 | 代表技术 |
|---------|---------|---------|
| 数据分片 | 路由计算 | Redis Cluster、MyCat |
| 服务拆分 | RPC / 消息 | Dubbo、Kafka |
| 任务拆分 | 结果聚合 | MapReduce |

## 二、无状态设计

### 2.1 原则

> **状态集中管理，服务节点保持无状态。**

```
✅ 正确的做法：
请求 → 负载均衡 → 任意 Node → 共享存储（DB/Redis）
每个节点不保存用户状态

❌ 错误的做法：
请求 → 负载均衡 → Node A（保存了用户 session）
下次请求 → 负载均衡 → Node B（没有该 session）
```

### 2.2 哪些应该无状态

| 组件 | 状态 | 说明 |
|------|------|------|
| Web 服务器 | ❌ 无状态 | Session 放在 Redis |
| 微服务实例 | ❌ 无状态 | 可随意扩缩容 |
| API 网关 | ❌ 无状态 | 路由和限流逻辑 |
| 数据库 | ✅ 有状态 | 数据持久化在磁盘 |
| 缓存 | ✅ 有状态 | 数据在内存中 |

### 2.3 无状态的好处

```
1. 弹性伸缩
   任意节点可被替换
   新增节点立即提供服务

2. 故障恢复
   某节点宕机 → 流量切换到其他节点
   不影响业务连续性

3. 滚动升级
   逐个替换节点，不影响整体服务
```

## 三、读写分离

将读操作和写操作分离到不同的节点或不同的存储系统中。

### 3.1 数据库读写分离

```
主库（Write）：INSERT / UPDATE / DELETE
从库（Read）：SELECT

写请求 → Master → 同步 → Slave1 / Slave2 → 读请求
```

**关键挑战：主从延迟**

```java
// 写后立即读的问题
public User createUser(User user) {
    userDao.insert(user);          // 写入主库
    return userDao.findById(user.getId());  // 读到从库 → 可能延迟
}

// 解决方案：
// 1. 写后强制读主库
// 2. 前端延迟展示
// 3. 业务允许短暂的不一致
```

### 3.2 缓存 + 数据库读写分离

```
写：更新数据库 → 删除缓存
读：查缓存 → 未命中 → 查数据库 → 回填缓存

这是最常见的 Cache Aside 模式。
```

## 四、最终一致性

### 4.1 一致性模型

| 模型 | 说明 | 代价 |
|------|------|------|
| 强一致性 | 读一定读到最新写入 | 可用性降低，延迟增加 |
| 弱一致性 | 读可能读到旧数据 | 可用性高 |
| 最终一致性 | 过一段时间后一定能读到最新数据 | 折中方案 |

### 4.2 最终一致性的实现

```
最终一致性的常见实现：

1. 异步消息
   主操作成功后，发消息让下游异步处理
   消息中间件保证消息不丢

2. 定期对账
   定时任务检查两边数据是否一致
   不一致则补偿修复

3. 回调重试
   失败后重试，直到成功
   配合幂等性保证不会重复执行
```

## 五、幂等性

幂等性是指**同一个操作执行多次和执行一次的效果相同**。

### 5.1 需要幂等的场景

```
1. 网络超时后重试
2. MQ 消息重复消费
3. 前端重复提交
4. 支付回调重复通知
```

### 5.2 幂等性实现方案

```java
// 方案一：唯一键约束
@Insert("INSERT INTO orders(id, ...) VALUES(#{id}, ...) " +
        "ON DUPLICATE KEY UPDATE ...")  // 重复则忽略
public void createOrder(Order order);

// 方案二：状态机
// 订单状态流转：INIT → PAID → SHIPPED → COMPLETED
// 已经 PAID 的订单，再次收到支付回调时直接返回成功
public void processPayCallback(Long orderId) {
    Order order = orderDao.findById(orderId);
    if (order.getStatus() >= OrderStatus.PAID) {
        return;  // 已处理，直接返回
    }
    // 处理支付成功逻辑
}

// 方案三：去重表
// 使用 Redis SETNX 或数据库唯一键
public boolean tryProcess(String businessKey) {
    return redisTemplate.opsForValue()
            .setIfAbsent("dedup:" + businessKey, "1", Duration.ofHours(24));
}
```

## 六、服务降级与熔断

### 6.1 服务降级

```
有策略地放弃一些非核心功能，保证核心功能的可用性。

降级分类：
├── 限流降级：超过阈值后返回降级响应
├── 超时降级：外部依赖超时后返回默认值
├── 异常降级：外部依赖异常时返回缓存数据
└── 功能降级：暂时关闭非核心功能
```

```java
// 降级示例：推荐服务超时时返回空列表
public List<Product> recommendProducts(Long userId) {
    try {
        return recommendService.getRecommend(userId);
    } catch (TimeoutException e) {
        log.warn("推荐服务超时，返回默认推荐");
        return Collections.emptyList();  // 降级
    }
}
```

### 6.2 熔断

```
熔断器三种状态：
CLOSED（关闭）→ 正常运行，请求正常通过
OPEN（打开）  → 故障率过高，直接拒绝请求
HALF_OPEN（半开）→ 尝试放行少量请求，检测是否恢复

             故障率超阈值
  CLOSED ─────────────────▶ OPEN
    ↑                        │
    │    HALF_OPEN 成功     │
    │◀──────────────────────│
    │                        │
    │      超时后进入半开    │
    │◀───────────────────────│
```

```java
// 使用 Resilience4j 实现熔断
@CircuitBreaker(name = "userService", fallbackMethod = "getDefaultUser")
public User getUser(Long id) {
    return userService.findById(id);
}

public User getDefaultUser(Long id, Throwable t) {
    log.warn("用户服务熔断，返回默认用户", t);
    return new User(id, "default");
}
```

## 七、异步解耦

### 7.1 同步 vs 异步

```
同步调用：
服务A → 等待 → 服务B → 等待 → 服务C
        ↑ 整个链路中最慢的环节决定响应时间

异步消息：
服务A → 消息队列 → 服务B → 消息队列 → 服务C
      ↘ 立即返回，由 MQ 保证最终送达
```

### 7.2 异步解耦的好处

```
1. 削峰填谷
   秒杀请求 → MQ → 按能力消费
   不是一次性压垮下游

2. 故障隔离
   下游宕机 → MQ 积压 → 恢复后继续消费
   不需要上游感知

3. 异步编排
   订单创建 → 发短信 / 发邮件 / 更新库存 / 积分
   这些操作不需要同步等待
```

## 八、总结

### 分布式设计原则速记

| 原则 | 一句话 |
|------|--------|
| 分治 | 大问题拆小问题，分而治之 |
| 无状态 | 节点不保存状态，状态放共享存储 |
| 读写分离 | 读写操作分离到不同节点 |
| 最终一致性 | 不强求任何时候一致，保证最终一致 |
| 幂等性 | 同一操作执行多次和执行一次效果相同 |
| 降级熔断 | 有策略地放弃非核心功能 |
| 异步解耦 | 通过消息中间件解耦上下游 |

### 核心权衡

```
一致性 ←------------------→ 可用性
（越一致越慢）            （越快越可能不一致）

延迟 ←--------------------→ 吞吐量
（处理越快）              （批量越大）

强耦合 ←------------------→ 解耦
（开发简单）              （运维复杂）
```
